## 1. 概述

本文档提供渠道网络故障的标准化诊断和处理流程，确保故障响应一致、高效，最大限度减少业务中断时间。

## 2. 故障分级

| 级别 | 描述 | 响应时间 | 示例 |
|----|------|---------|------|
| `P0` | 渠道完全不可用，所有交易失败 | < 5 分钟 | 链路中断、DNS 故障 |
| `P1` | 部分故障，错误率 >10% | < 15 分钟 | 超时激增、丢包 |
| `P2` | 性能下降，延迟升高 | < 30 分钟 | 响应变慢、间歇性错误 |
| `P3` | 轻微问题，无业务影响 | < 2 小时 | 证书即将过期、监控告警 |

## 3. 初始分诊（前 5 分钟）

### 3.1 确认问题

```
□ 查看监控大盘告警
□ 从多个来源验证（排除单点探测误报）
□ 确认业务影响：交易成功率、错误码
□ 识别受影响的渠道和范围（全部流量 vs 部分流量）
```

### 3.2 快速健康检查

```bash
# DNS 解析
dig <渠道域名>
nslookup <渠道域名>

# 连通性
ping <渠道IP>
telnet <渠道IP> <端口>
curl -v https://<渠道端点>/health

# SSL/TLS 检查
openssl s_client -connect <渠道IP>:443 -servername <渠道域名>

# 路由追踪
traceroute <渠道IP>
mtr <渠道IP>
```

### 3.3 判断故障类型

| 现象 | 可能原因 | 下一步 |
|------|---------|--------|
| DNS 解析失败 | DNS 配置错误 / 服务商故障 | → 4.1 节 |
| 连接被拒绝 | 服务宕机 / 防火墙拦截 | → 4.2 节 |
| 连接超时 | 网络路径问题 / 路由异常 | → 4.3 节 |
| SSL 握手失败 | 证书问题 | → 4.4 节 |
| HTTP 5xx 错误 | 上游服务故障 | → 4.5 节 |
| 高延迟 / 丢包 | 网络拥塞 / 链路质量下降 | → 4.6 节 |

## 4. 诊断与处理

### 4.1 DNS 故障

**现象：** NXDOMAIN、SERVFAIL、解析返回错误 IP

**处理步骤：**
1. 验证 DNS 记录：`dig <域名> @8.8.8.8`
2. 检查 DNS 传播是否完成：`dig <域名> @<权威NS>`
3. 与预期 IP/CNAME 对比
4. 如果错误：检查 DNS 服务商控制台是否有近期变更
5. 如果服务商故障：临时切换备用 DNS 或更新本地 hosts

### 4.2 连接被拒绝

**现象：** Connection refused，端口未开放

**处理步骤：**
1. 确认目标端口是否正确
2. 从不同源 IP 测试（排除 IP 白名单问题）
3. 联系渠道方确认服务状态
4. 检查我方出口 IP 是否变更（NAT 网关、EIP）
5. 检查安全组 / 防火墙规则

### 4.3 连接超时

**现象：** Connection timed out，无响应

**处理步骤：**
1. Traceroute 定位丢包节点
2. 判断问题在我方（出站）还是渠道方（入站）
3. 检查路由表和 NAT 网关状态
4. 查看 ISP / 云服务商网络状态页
5. 如有备用网络路径，尝试切换测试

### 4.4 SSL/TLS 故障

**现象：** SSL handshake failure、证书过期、证书不匹配

**处理步骤：**
1. 检查证书有效期：`openssl s_client -connect <主机>:443 | openssl x509 -noout -dates`
2. 验证证书链：`openssl s_client -connect <主机>:443 -showcerts`
3. 检查 SNI 是否正确
4. 验证 TLS 版本兼容性
5. 检查证书是否近期更新/变更

### 4.5 HTTP 错误（5xx）

**现象：** 502 Bad Gateway、503 Service Unavailable、504 Gateway Timeout

**处理步骤：**
1. 判断错误来源是 CDN/代理 还是源站
2. 检查响应头确认错误来源（X-Cache、Via、Server）
3. 查看渠道方状态页
4. 如可能，绕过 CDN 直连源站测试
5. 检查请求内容是否有问题（大小、格式、编码）

### 4.6 高延迟 / 丢包

**现象：** 响应缓慢、间歇性失败、抖动

**处理步骤：**
1. MTR 定位丢包节点：`mtr -r -c 100 <渠道IP>`
2. 判断问题是持续性还是间歇性
3. 与基线延迟对比
4. 检查我方带宽使用率
5. 检查特定跳点是否存在网络拥塞

## 5. 升级矩阵

| 经过时间 | 动作 |
|---------|------|
| 0-5 分钟 | 值班工程师开始分诊 |
| 5-15 分钟 | P0 级别：通知团队负责人，开始缓解 |
| 15-30 分钟 | 未解决：升级至网络团队 / 云服务商 |
| 30-60 分钟 | P0 未解决：升级至管理层，启动容灾方案 |
| 1 小时+ | 向相关方发送事件通报 |

## 6. 临时缓解措施

| 措施 | 适用场景 |
|------|---------|
| 切换备用渠道 | 主渠道完全不可用 |
| 切换备用链路/ISP | 网络路径故障 |
| 绕过 CDN 直连源站 | CDN 层故障 |
| 启用熔断器 | 防止级联故障 |
| 限流 | 部分降级，保护剩余容量 |
| DNS 故障转移 | DNS 服务商故障 |

## 7. 沟通模板

### 内部通知
```
[P0/P1] 渠道网络故障 - <渠道名称>
时间：<时间戳>
影响：<业务影响描述>
状态：排查中 / 缓解中 / 已恢复
当前动作：<正在执行的操作>
预计恢复：<预计恢复时间>
```

### 取证要求
>故障处理过程中需截图留证，用于事后复盘和对外沟通：

| 类型 | 内容 | 说明 |
|------|------|------|
| 监控截图 | 监控大盘关键指标 | 成功率、错误率、延迟曲线等异常时段截图 |
| 可达探测 | ping / traceroute / mtr 结果 | 截图或文本记录，标注丢包节点 |
| 端口探测 | telnet / curl 结果 | 记录连接状态（成功/拒绝/超时） |
| 日志截图 | 网关/应用错误日志 | 关键错误信息截图 |
| 时间线 | 各操作时间点 | 发现时间、处理时间、恢复时间 |

>注意: 截图需包含时间戳，便于事后对齐时间线。

### 渠道方升级
```
主题：[紧急] 连接贵方端点异常

我方在连接贵方端点时遇到异常：
- 端点：<url>
- 错误：<错误描述>
- 开始时间：<时间戳>
- 我方出口 IP：<ip>

请协助排查，谢谢。
```

## 8. 事后处理

- 记录事件时间线
- 确定根本原因
- 如发现监控盲区，补充告警规则
- 如遇到新场景，更新本 SOP
- P0/P1 级别安排复盘会议
- 落实预防措施
